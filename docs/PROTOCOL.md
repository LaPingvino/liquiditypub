# LiquidityPub Federation Protocol

**Version:** 0.2-draft
**Status:** Living specification — not yet stable
**Date:** July 2026
**Repository:** https://github.com/LaPingvino/liquiditypub
**Rationale:** [DESIGN.md](DESIGN.md) — read it first; this document specifies *what*,
that one explains *why*.
**Scope:** This protocol governs the **contact surface only** — identity, envelopes,
transport, contacts, transfers, checkpoints. A node's internals (ledger structure,
issuance mechanics, membership policy) are deliberately out of scope: weird internal
implementations are welcome and require no protocol changes. Sections marked
*(reference model)* describe the recommended internal shape and are non-normative;
peers rely only on published interface facts and price what they observe.

---

## 0. Changes from v0.1

- **Object signatures replace HTTP Signatures.** Envelopes are self-signed JSON
  (ed25519 over JCS canonical form), so they survive CDNs, static files, polling, and
  git replication. Transport-level signatures are gone.
- **Pull federation is the mandatory baseline; push is an optimization.** Every node
  publishes per-peer outboxes as static-servable JSON; an active inbox is a capability.
- **The 0–100 trust scalar is replaced by contacts**: each node holds a real wallet on
  the peer's ledger, and the two reciprocal reserves price exchanges as a built-in
  per-contact market (constant-product), so relative value floats per contact. There is
  no pegged mode: a node that wants a stable rate defends it with its own reserve
  policy (§8.4), currency-board style — imitation is node business, not protocol.
- **`exchange_request`/`exchange_confirm` are replaced by a transfer state machine**
  (`propose → accept → commit → receipt`) with expiry, idempotency, and abort.
- **Reserves and reconciliation** are specified: node wallets, ordered contact
  operations, channel hashes, signed checkpoints.
- **Universal Dividend issuance is first-class**, with published monetary parameters;
  UD parity is the recommended seed ratio and display anchor (not an enforced rate).
- **Ledger publication** is specified: hash-linked log, transparency levels.

## 1. Terminology

| Term | Meaning |
|---|---|
| **Node** | One community currency instance, identified by an HTTPS origin (e.g. `https://riverside.example`). |
| **Member** | An account holder at a node, addressed `name@host` (e.g. `alice@riverside.example`). |
| **Contact** | The bilateral relationship between two nodes: reciprocal node wallets forming a pool. |
| **Node wallet** | The account a peer node holds on this node's ledger (`node:<host>`); its balance is that peer's **reserve** here. |
| **Pool** | The pair of reciprocal reserves of a contact, pricing exchanges by constant product. |
| **Transfer** | A cross-node payment executed over a contact. |
| **UD** | Universal Dividend — the equal periodic issuance to every member. |
| **Channel** | The ordered message stream from one node to one peer (`from → to`). |

Key words MUST / SHOULD / MAY are as in RFC 2119.

## 2. Conventions

- **Amounts** are integers in **micro-units**: 1 currency unit = 1,000,000 micro-units.
  Amounts MUST NOT exceed 2⁵³−1 (JSON number safety).
- **No floating point anywhere.** Every number in protocol JSON is an integer;
  fractional quantities (weights, rates-implied values) are expressed in millionths
  (ppm / micro-weights). Intermediate arithmetic (e.g. pool pricing products) MUST be
  computed exactly (128-bit or arbitrary precision) before flooring.
- **Canonicalization:** JSON Canonicalization Scheme (JCS, RFC 8785). Because protocol
  JSON contains only integers, strings, booleans, null, arrays, and objects, JCS
  reduces to: UTF-8, no insignificant whitespace, object keys sorted by code point,
  integers in plain decimal, minimal string escaping.
- **Signatures:** ed25519 (RFC 8032). **Hashes:** SHA-256.
  Both encoded as base64url without padding. Public keys are the raw 32 bytes.
- **Timestamps:** RFC 3339 UTC with `Z` suffix.
- **Ids:** `urn:uuid:` URNs, generated by the message creator.
- **Rates** are integer fractions `{"n": <int>, "d": <int>}`;
  `dst_amount = floor(src_amount × n / d)`.

## 3. Node identity document

Every node MUST serve, as a static-servable document:

```
GET /.well-known/liquiditypub
Content-Type: application/json
```

```json
{
  "liquiditypub": "0.2",
  "node": {
    "name": "Riverside",
    "description": "A local libre currency for the Riverside neighborhood",
    "base": "https://riverside.example"
  },
  "currency": {
    "name": "River Credits",
    "symbol": "ʀ",
    "micro_units": 1000000
  },
  "issuance": {
    "type": "ud",
    "c_period_ppm": 274,
    "ud_period": "P1D",
    "current_ud": 1200000
  },
  "keys": [
    {
      "id": "#nk1",
      "alg": "ed25519",
      "public_key": "<base64url, 32 bytes>",
      "created": "2026-07-01T00:00:00Z",
      "revoked": null
    }
  ],
  "endpoints": {
    "inbox": "/lp/inbox",
    "outbox": "/lp/outbox/{peer-host}.json",
    "checkpoint": "/lp/checkpoint.json",
    "log": "/lp/log/"
  },
  "capabilities": ["pull", "push", "member-keys", "sealed-outbox"],
  "transparency": "pseudonymous",
  "peers": ["https://hilltop.example"],
  "stats": {
    "active_members": 57,
    "ud_weight_total": 60500000,
    "money_supply": 123400000000
  }
}
```

Notes:

- `issuance` publishes the node's **actual monetary policy as auditable fact**; peers
  use it to price trust (DESIGN §5–6). For `type: "ud"`: `c_period_ppm` is the
  per-period supply growth in parts-per-million (274/day ≈ 10%/year), `ud_period` is
  an ISO 8601 duration, `current_ud` is the present standard-weight dividend in
  micro-units (§10). Other types: `"periodic_fixed"` (fixed amount per period) and
  `"manual"` — or anything else the node truthfully publishes.
- **Key rotation:** a new key is added to `keys` and announced with a `key.announce`
  message signed by a currently valid old key. Verifiers MUST accept any listed,
  non-revoked key. Revocation sets `revoked` to a timestamp.
- **Delegation (clearing houses):** a node MAY list an **operator key** held by a
  clearing house that runs its live operations, alongside the community's own master
  key, with `endpoints` pointing at the operator's infrastructure. The identity
  document on the node's own domain remains the root of trust: revoking the operator
  key and repointing endpoints switches (or fires) the clearing house. The node's own
  host SHOULD serve checkpoint and log mirrors as exit insurance. Key entries reserve
  an OPTIONAL `policy` field for the v0.3 covenant DSL (a master-key-signed predicate
  program scoping what the key may authorize); v0.2 implementations MUST treat a key
  bearing an unrecognized `policy` as valid-but-unscoped only if the master key also
  signs the operation, and otherwise reject — fail closed.
- `capabilities`: `"pull"` is mandatory and listed for explicitness; `"push"` means the
  inbox endpoint is live; `"member-keys"` means member signatures appear in the log
  (sovereign profile); `"sealed-outbox"` means peer-addressed payloads are encrypted.
- `transparency`: `"public"`, `"pseudonymous"`, or `"peers"` (§9.3). Checkpoints are
  public in every mode.
- `peers`: OPTIONAL list of contact peers the node chooses to reveal. First contact is
  always manual/out-of-band; further discovery works by crawling outward from existing
  connections through these lists.

## 4. Message envelope

Every federation message is a signed envelope:

```json
{
  "lp": "0.2",
  "id": "urn:uuid:9c5b1c9e-…",
  "type": "transfer.propose",
  "from": "https://riverside.example",
  "to": "https://hilltop.example",
  "seq": 42,
  "re": null,
  "created": "2026-07-02T12:00:00Z",
  "payload": { },
  "sig": {
    "key": "https://riverside.example/.well-known/liquiditypub#nk1",
    "alg": "ed25519",
    "value": "<base64url>"
  }
}
```

- `seq`: strictly increasing integer per **channel** (`from → to`), starting at 1.
- `re`: optional — the `id` of the message this one answers.
- **Signing:** remove the `sig` member entirely, canonicalize the remaining envelope
  with JCS, sign the resulting UTF-8 bytes with ed25519. (Test vectors:
  `conformance/vectors/envelope_sign.json`.)
- **Verification order:** (1) fetch sender identity doc, resolve `sig.key`;
  (2) verify signature; (3) reject if `id` was already processed (idempotent replay:
  re-answer with the original response); (4) reject if `seq` ≤ last processed seq for
  the channel and `id` differs (out-of-order/forgery); (5) reject if `created` is more
  than 7 days old or 1 hour in the future; (6) dispatch by `type`.

Unknown `type` → respond with an `error` message (`re` set, `payload.code:
"unknown-type"`). Errors are themselves envelopes:

```json
{ "type": "error", "re": "urn:uuid:…", "payload": { "code": "limit-exceeded", "detail": "…" } }
```

## 5. Transport bindings

The same envelopes travel over either binding. **Pull is the mandatory baseline**
(every conforming node can be fully static on the read side); **push is an
optimization** for low latency.

### 5.1 Pull (mandatory)

- A node maintains, for each contact peer, an outbox document at
  `endpoints.outbox` with `{peer-host}` substituted (e.g. `/lp/outbox/hilltop.example.json`):
  a JSON array of envelopes addressed to that peer, ordered by `seq`.
- Peers SHOULD poll each other's outboxes at an agreed cadence (see §12 profile
  guidance) and process envelopes in `seq` order.
- **Acknowledgement & pruning:** each node's checkpoint (§8.3) publishes
  `last_seq_processed` per peer. A sender MAY prune outbox entries with
  `seq ≤` the peer's published `last_seq_processed`.
- **Confidentiality:** outboxes are public files. Nodes with the `sealed-outbox`
  capability (EXPERIMENTAL) encrypt each `payload` to the peer's node key
  (X25519 sealed box derived from the ed25519 key); the envelope frame stays
  cleartext so `seq`/pruning still work. The normative sealing rule is deferred until
  field experience with v0.2 deployments; static mirrors SHOULD simply omit outboxes.

### 5.2 Push (capability `"push"`)

```
POST /lp/inbox
Content-Type: application/json
```

Body: one envelope. Response: `202` with `{"status":"accepted"}` after signature
verification, or `400`/`403` with an error body. Substantive replies are delivered
asynchronously as new messages (pushed back, or placed in the outbox for pull) —
the HTTP response is transport acknowledgement only.

A node with `push` capability MUST still publish outboxes and checkpoints, so pull-only
peers can federate with it.

## 6. Contacts

A contact is opened and maintained with these messages:

| Type | Direction | Effect |
|---|---|---|
| `contact.propose` | either | Propose opening a contact (seed offer) or changing its terms. |
| `contact.accept` | responder | Activate the contact; both sides apply their seed entries. |
| `contact.update` | either | Unilateral changes to own parameters (e.g. a liquidity top-up offer). |
| `contact.close` | either | Freeze new operations (reserves survive until withdrawn by consent). |

### 6.1 `contact.propose` payload

```json
{
  "contact_id": "urn:uuid:…",
  "my_seed": 500000000,
  "note": "Neighboring town, weekly market overlap"
}
```

- `my_seed`: the amount of **the proposer's own currency** it will place in the
  responder's node wallet on the proposer's ledger upon acceptance. The responder's
  `my_seed` in `contact.accept` completes the pool; **the seed ratio is the opening
  rate**. Between UD nodes, UD parity is the RECOMMENDED default ratio. Whether seeds
  are freshly issued or paid from a treasury account is node policy (it is an internal
  transaction either way).

On acceptance, both nodes append their seed entries and the contact's operation
counter starts at `op_seq = 0`.

### 6.2 Pricing

Let `R_src` = the reserve of the *destination* node held at the source
node (its wallet balance there), and `R_dst` = the reserve of the *source* node held at
the destination. For a transfer of `src_amount`:

```
dst_amount = floor( R_dst × src_amount / (R_src + src_amount) )
```

(Constant product: `R_src` grows by `src_amount`, `R_dst` shrinks by `dst_amount`,
`k = R_src × R_dst` is preserved up to rounding, which MUST round in the pool's favor —
`floor` on `dst_amount` guarantees `(R_src + src)(R_dst − dst) ≥ k`.) The product
`R_dst × src_amount` can exceed 2⁵³; intermediates MUST be exact (§2). Proposals whose
computed `dst_amount` is 0 are invalid (`error: dust`). Reserves — and therefore
prices — are deterministic functions of the contact's committed operation history,
ordered by `op_seq` (§8.2). (Test vectors: `conformance/vectors/pool_pricing.json`.)

There is deliberately no fixed-rate mode: a node that wants a stable rate against a
peer defends it with its own reserve policy — topping up or withdrawing liquidity via
§8.4 to hold the pool near a target — currency-board style. How (and whether) a node
does this is its own business.

### 6.3 Serialization

A contact processes **at most one operation at a time**: a node MUST reject a
`transfer.propose` on a contact that already has an operation in the
ACCEPTED-but-not-yet-SETTLED state (`error: busy`). Every committed operation
increments `op_seq`. Proposals pin the `op_seq` they priced against; a commit whose
`op_seq` is no longer current is rejected (`error: stale-pool`) and MUST be re-proposed
at the new state. At community scale this serialization costs nothing and removes all
concurrent-pricing ambiguity.

## 7. Transfers

### 7.1 State machine

```
            propose            accept              commit             receipt
  (payer) ──────────▶ PROPOSED ────────▶ ACCEPTED ─────────▶ COMMITTED ───────▶ SETTLED
                          │                  │                    │
                          │ reject/expire    │ abort/expire       │ (receipt lost:
                          ▼                  ▼                    │  re-send commit,
                       REJECTED           ABORTED                 ▼  idempotent)
                                                               SETTLED
```

- The **payer's node** proposes. `expires` bounds PROPOSED and ACCEPTED.
- On **accept**, the receiving node MUST hold the payout (`dst_amount` from the
  sender's node wallet) until commit or expiry — an accept is a promise that commit
  cannot fail, and it locks the contact (§6.3).
- On **commit**, the sender has irrevocably appended its ledger entries; the receiver
  MUST append its own and answer with `transfer.receipt`. A duplicate `transfer.commit`
  (retry) MUST be answered with the same receipt, not re-applied.
- `transfer.abort` is valid only before the sender's commit is appended.
- Sender crash after commit, receiver silence, lost receipts — all resolve by
  idempotent retry of `commit`; anything that slips past is caught by checkpoint
  reconciliation (§8.3), and the damage is bounded by the credit limit.

### 7.2 Messages

`transfer.propose` payload:

```json
{
  "transfer_id": "urn:uuid:…",
  "contact_id": "urn:uuid:…",
  "op_seq": 12,
  "from_member": "alice@riverside.example",
  "to_member": "bob@hilltop.example",
  "src_amount": 10000000,
  "dst_amount": 19607843,
  "note": "veggie box",
  "expires": "2026-07-02T12:10:00Z"
}
```

The receiver validates: contact active and not busy (§6.3); `op_seq` current; amounts
exactly reproduce the pricing formula (§6.2) from its own view of the reserves;
`to_member` exists and is active. Then
`transfer.accept {transfer_id}` or `transfer.reject {transfer_id, code, detail}`.

`transfer.commit` payload:

```json
{
  "transfer_id": "urn:uuid:…",
  "entry": { "log_seq": 1042, "log_hash": "<hash of sender's log entry>" }
}
```

`transfer.receipt` mirrors it with the receiver's `entry`. Both proofs are checkable
against the published logs (per transparency level).

### 7.3 Ledger effect

For a transfer of `src_amount` ʀ → `dst_amount` ʜ over the Riverside–Hilltop contact:

| Ledger | Entries |
|---|---|
| Riverside (sender) | `m:<alice> −src_amount`, `node:hilltop.example +src_amount` |
| Hilltop (receiver) | `node:riverside.example −dst_amount`, `m:<bob> +dst_amount` |

Both legs are ordinary internal payments; the pool mechanism lives entirely in the
validation rules, not in the ledger structure.

### 7.4 Expiry guidance

`expires` SHOULD be ≥ 4× the slower party's polling interval. Recommended defaults:
10 minutes when both nodes push; 1 hour for pull-polling pairs at the recommended
15-minute cadence. Every operating node is live (§12), so there is no slow profile.

## 8. Reserves and reconciliation

### 8.1 Node wallets and exposure

Each node keeps one **node wallet** per contact peer, `node:<host>`, denominated in
**its own currency**, holding that peer's reserve. Definitions:

- My **exposure** to a peer = my reserve held at *them* (their `node:<my-host>`
  balance, in their currency) — the most a confiscating or vanishing peer can take
  from me. It is bounded by what I chose to seed plus accumulated flows, and it is
  mirrored: the peer's reserve at me is symmetric collateral.
- Node wallets MUST NOT go negative; they change *only* through contact operations
  (seeds, transfers, adjustments), so reserves are a deterministic function of the
  operation history.

### 8.2 Invariants

1. **Local conservation** (per node, always): `SUM(all ledger entry amounts) = 0`.
2. **Contact consistency**: both sides have applied the same committed operations in
   the same order. Tracked as a **channel hash**, identically computable on both sides:

   ```
   root₀ = SHA-256( UTF-8(contact_id) )
   rootₙ = SHA-256( rootₙ₋₁ ‖ JCS([op_type, op_id, src_amount, dst_amount]) )
   ```

   where `rootₙ₋₁` is the raw 32 bytes and the JCS array is UTF-8 (e.g.
   `["transfer","urn:uuid:…",10000000,19607843]`), over committed operations in
   `op_seq` order (seeds are operation 0, encoded as
   `["seed", contact_id, seed_src, seed_dst]` with each side's seed in the proposing
   and responding currency respectively). Same root ⇒ same reserves ⇒ same price, by
   construction. (Test vectors: `conformance/vectors/channel_hash.json`.)
3. **Pool invariant**: trades preserve `k = R_a × R_b` up to pool-favoring rounding;
   only seeds and consensual adjustments (§8.4) change `k`.

### 8.3 Checkpoints

Every node MUST publish a signed checkpoint at `endpoints.checkpoint`, refreshed at
least once per `ud_period` (or daily for non-UD nodes):

```json
{
  "lp": "0.2",
  "type": "checkpoint",
  "node": "https://riverside.example",
  "created": "2026-07-02T12:00:00Z",
  "log_seq": 1042,
  "log_hash": "<head hash>",
  "money_supply": 123400000000,
  "member_count": 57,
  "current_ud": 1200000,
  "contacts": [
    {
      "peer": "https://hilltop.example",
      "contact_id": "urn:uuid:…",
      "peer_reserve_here": 510000000,
      "op_seq": 13,
      "channel_root": "<hash>",
      "last_seq_processed": 41
    }
  ],
  "sig": { "key": "#nk1", "alg": "ed25519", "value": "…" }
}
```

`peer_reserve_here` is the peer's node-wallet balance on this ledger; each side of the
pool is thus published by the node that custodies it, and anyone can compute the
current rate from the two checkpoints. Peers MUST compare `channel_root` and `op_seq`
on every poll; divergence freezes the contact for new operations until resolved (out of
band — the histories are signed, so the discrepancy is attributable).
`last_seq_processed` doubles as the pull acknowledgement (§5.1).

### 8.4 Reserve adjustments

Consensual changes to reserves outside pricing — liquidity top-ups, withdrawals,
recording out-of-band settlement (goods, fiat, forgiveness) — use
`reserve.adjust {contact_id, op_seq, adjust_id, my_delta, memo}` answered by
`reserve.accept`. Each side appends its entry (counterpart on its `treasury` or
`issuance` account, per policy); the adjustment enters the channel hash as an
operation. Adjustments are how a depleted pool is refilled — and the primitive with
which a node implements any rate policy it likes, including defending a peg.

## 9. Ledger *(reference model)*

Non-normative except §9.3: what the contact surface requires is that contact
operations follow §6–8 and that published claims (checkpoints, stats, transparency
promises) are honored and internally consistent. The shape below is the recommended
way to make that easy and auditable.

### 9.1 Structure

- Double-entry: every transaction's entries sum to zero. Integer micro-units only.
- Account id namespaces: `m:<opaque>` members, `node:<host>` peer node wallets,
  `fund:<opaque>` community funds, and the `issuance` / `treasury` system accounts.
- Transaction types: `issuance.ud`, `issuance.grant`, `payment` (internal),
  `transfer.out`, `transfer.in`, `reserve.adjust`.

### 9.2 The log

The canonical ledger is an append-only, hash-linked log:

```json
{
  "seq": 1042,
  "prev": "<hash of entry 1041>",
  "tx": {
    "id": "urn:uuid:…",
    "type": "transfer.out",
    "ref": "<transfer_id>",
    "created": "2026-07-02T12:03:00Z",
    "entries": [
      { "account": "m:9f2c", "amount": -10000000 },
      { "account": "peer:hilltop.example", "amount": 10000000 }
    ]
  },
  "member_sig": null,
  "hash": "<SHA-256 of JCS(entry without hash)>"
}
```

- `member_sig` carries the paying member's signature over `tx` in the sovereign
  profile (`member-keys` capability); `null` under custodial operation. The formats are
  otherwise identical — custodial nodes can upgrade in place.
- The node does not sign every entry; it signs **checkpoints**, which chain to
  everything via `log_hash`.
- Published as static pages under `endpoints.log` (e.g. `/lp/log/page-21.json`, fixed
  page size, plus `/lp/log/head.json`).

### 9.3 Transparency levels

| Level | Log visibility | Account ids | Checkpoints |
|---|---|---|---|
| `public` | public | may be real usernames | public |
| `pseudonymous` *(recommended)* | public | opaque (`m:9f2c`); mapping private to the node | public |
| `peers` | contact peers only | opaque | **public** |

Checkpoints are public in every mode — money supply, member count, UD value, and
clearing positions are the minimum the trust model needs (DESIGN §6, §7).

## 10. Issuance *(reference model)*

- `ud`: dividend recipients receive `weight × current_ud` each `ud_period`
  (`tx.type: "issuance.ud"`, counterpart on `issuance`). Every active member is a
  recipient with weight 1 by default; a node MAY add further recipients with
  configured multipliers — community project funds (`fund:` accounts), even peer node
  wallets (a built-in liquidity drip to a contact). **How a node runs its currency is
  its own business**; what the protocol requires is that the aggregate is auditable:
  the identity document publishes `ud_weight_total` alongside `active_members`, and
  checkpoints publish money supply, so peers can always see effective dilution and
  compensate through exchange values and trust — or stop accepting the coin.
- Weights are integer **micro-weights** (default member weight 1,000,000);
  `ud_weight_total` is their sum. The reference formula, per `ud_period`, with
  `c_period_ppm` the per-period growth rate in parts-per-million:

  ```
  ud_base       = floor( money_supply × c_period_ppm / ud_weight_total )
  recipient_ud  = floor( ud_base × weight / 1000000 )
  ```

  `ud_base` is the dividend of a standard-weight member and is what `current_ud`
  publishes. This is one clean discretization, not a mandate — experimenting with the
  arithmetic is part of the project's purpose (DESIGN §5); nodes publish whatever they
  actually run (Ğ1 compatibility is a non-goal). (Test vectors:
  `conformance/vectors/ud_reference.json`.)
- `periodic_fixed`: fixed amount per recipient per period.
- `manual`: operator-triggered grants (`issuance.grant`).
- Issuance MUST be driven by a scheduler (cron, CI schedule, runtime timer) — never by
  request handling. Missed periods MUST be issued retroactively on next run, stamped
  with their nominal period.

## 11. Member addressing and lookup

- Members are `name@host`; `name` matches `[a-z0-9_]{1,32}` after lowercasing.
- `member.lookup {member}` → `member.result {member, found, display_name?}` (with `re`
  set). Nodes MUST NOT answer lookups except over an active contact, and SHOULD
  omit `display_name` unless node policy allows it.

## 12. Deployment profiles

| | A — active server | B — serverless | C — HTML-only via clearing house |
|---|---|---|---|
| Example stack | Go binary + SQLite; PHP + SQLite | CF Pages Functions + D1 + cron; GAE + Firestore | Static host (identity doc + mirrors) + operator key at a clearing house |
| Pull binding | MUST | MUST | served live by the clearing house |
| Push inbox | SHOULD | SHOULD | via the clearing house |
| Poll cadence (SHOULD) | ≤ 15 min | ≤ 15 min | (clearing house cadence) |
| Issuance driver | timer | cron trigger | clearing house scheduler |
| Write path | server process | function, single-writer | clearing house, single-writer per hosted node |
| Own-host duties | everything | everything | identity document (root of trust), checkpoint + log mirrors (exit insurance) |

A **clearing house** is a Profile A/B node operating one or more thin nodes under
delegated operator keys (§3). Hosted nodes are full protocol citizens — own identity,
own currency, own contacts — and peers need not know or care that they are hosted.
Transfers between nodes hosted at the same clearing house MAY clear internally
(identical ledger effects, no federation transport). All profiles publish the same
identity doc, checkpoints, and (per transparency) logs; every operating node is live,
so transfer latency is uniform across profiles.

## 13. Security considerations

- **Bounded extraction** (the central property): a malicious node's maximum gain is the
  sum of the reserves peers placed with it, minus
  its own reserves forfeited at those peers — and pool pricing decelerates any drain,
  since dumping a currency collapses its own rate. Every other mechanism — published
  issuance facts, checkpoints, channel hashes — exists to help peers size reserves
  well, detect misbehavior early, and stop accepting a failed coin.
- **Replay & ordering:** message `id` dedup + per-channel `seq` + the `created` window.
  Processed-id retention MUST cover the `created` acceptance window (7 days).
- **What signatures do and don't give you:** object signatures prove origin and
  integrity, including from static mirrors. They do not prevent a node from censoring
  its own members (sovereign member keys prevent *forgery*, not censorship) or from
  lying by omission — which is what checkpoints and channel hashes are for.
- **Key compromise:** rotate via `key.announce` + identity doc update; peers SHOULD
  re-verify out of band on rotation. A stolen node key can forge messages but cannot
  raise the *peers'* limits — exposure stays bounded.
- **Transport:** HTTPS is assumed; object signatures make it a privacy layer, not an
  authenticity layer. Sealed outboxes protect payload confidentiality on static hosts.
- **DoS:** inboxes SHOULD rate-limit unauthenticated traffic; pull nodes are as
  DoS-resistant as their static host.
- **Clocks:** the 7-day/1-hour window tolerates bad clocks; `seq` provides ordering, so
  clock skew cannot reorder money.

---

## 14. Versioning and capabilities

- `lp` / `liquiditypub` fields carry the protocol version. 0.x versions may break
  compatibility; nodes MUST reject envelopes with a higher major version than they
  implement.
- New message types and capabilities are additive; unknown capability strings MUST be
  ignored, unknown message types answered with `error: unknown-type`.
